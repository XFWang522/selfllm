# 大模型基座模型团队（基模团队）工作分类全景图

> 基于对 OpenAI、Anthropic、Google DeepMind、Meta FAIR、DeepSeek、字节 Seed 等团队的调研，系统梳理基模团队的工作类别。

---

## 总览：九大工作类别

```
┌─────────────────────────────────────────────────────────────────────┐
│                     基座模型团队工作全景                              │
├─────────┬─────────┬─────────┬─────────┬─────────┬─────────────────┤
│         │         │         │         │         │                 │
│ ① 数据  │ ② 预训练│ ③ 后训练│ ④ 推理  │ ⑤ 多模态│ ⑥ 模型架构      │
│   工程  │         │  /对齐  │  模型   │         │   研究         │
│         │         │         │         │         │                 │
├─────────┼─────────┼─────────┴─────────┴─────────┴─────────────────┤
│         │         │                                               │
│ ⑦ 评估  │ ⑧ 训练  │ ⑨ 推理部署与模型压缩                          │
│   体系  │  基础设施│                                               │
│         │         │                                               │
└─────────┴─────────┴───────────────────────────────────────────────┘
```

**运作节奏差异**（引自 interconnects.ai 对各大实验室的分析）：
- **预训练团队**：低频大跑（一年 1-3 次大规模训练），每次跑数周到数月
- **后训练团队**：高频迭代（每周甚至每天多轮实验），迭代速度快
- **数据团队**：持续运转，为预训练和后训练持续供给数据
- **基础设施团队**：7×24 保障，是所有团队的底座

## ① 数据工程（Data Engineering）

**别名**：Data Curation / Data Pipeline / 数据团队

**核心职责**：构建、清洗、筛选和管理万亿级 token 的训练数据。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **数据采集** | Common Crawl 解析、网页爬取、书籍/论文/代码获取、许可证合规 | FineWeb2 (2025) 覆盖 1000+ 语言、20TB 多语言数据 |
| **数据清洗** | HTML 去标签、去噪、去广告、文本提取、格式标准化 | Blu-WERP pipeline，比 DCLM 高 4.0% |
| **质量过滤** | 分类器质量评分、困惑度过滤、信息密度评估、有害内容过滤 | 分类器 > 启发式规则已成共识；FineWeb 证明每步过滤都对下游有效 |
| **去重** | 精确去重（hash）、模糊去重（MinHash+LSH）、文档级/段落级去重 | NeMo Data Curator：千核级分布式去重 |
| **数据配比** | 不同来源（网页/书籍/代码/学术/对话）的混合比例调优 | Scaling law 指导配比；DoReMi 自动优化权重 |
| **Tokenization** | 分词器设计与训练（BPE/SentencePiece）、多语言 tokenizer | Llama 3 的 128K vocab tokenizer，压缩率提升 15% |
| **去污染** | 确保训练集不包含评估集数据，防止 benchmark 泄漏 | 成为标配流程；TULU 3 强调 decontamination |
| **合成数据生成** | 用强模型生成训练数据、知识蒸馏数据、self-play 数据 | IOA 框架：学生模型保留 94.7% 教师性能；PerSyn 个性化路由 |
| **版权与合规** | 数据许可证审查、PII 清除、版权内容识别与过滤 | 越来越重要，NYT v. OpenAI 诉讼推动行业合规化 |

**团队规模参考**：通常占基模团队 15-20%，是最容易扩展的团队之一。

**典型工具栈**：Spark/Dask（分布式处理）、NeMo Curator、datatrove、fastText（语言检测）、MinHash

## ② 预训练（Pre-training）

**别名**：Base Model Training / 基座训练

**核心职责**：在万亿级 token 上从零训练（或继续训练）基座模型。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **训练执行** | 大规模分布式训练的运行、监控、故障恢复、断点续训 | DeepSeek V3：2048 H800，288 万 GPU-hours，仅 $5.6M |
| **超参数选择** | Learning rate、batch size、warmup、decay schedule、weight decay | SmolLM3 Playbook：详细记录了真实决策过程和失败案例 |
| **Scaling Law 研究** | 模型大小 vs 数据量 vs 计算量的最优配比、损失预测 | Chinchilla Law → 过时；新共识倾向过度训练（overtrain）小模型 |
| **训练稳定性** | Loss spike 处理、梯度爆炸/消失、NaN 检测与修复 | MoFa (2025)：统一性能建模，预测瓶颈 |
| **混合精度训练** | BF16/FP16/FP8 混合精度策略、loss scaling | DeepSeek V3 首创大规模 FP8 训练无性能损失 |
| **持续预训练** | 在已有基座上继续预训练新数据（知识更新、领域适配） | 比从零训练成本低 10-100 倍，成为主流做法 |
| **数据课程策略** | 训练过程中动态调整数据配比和难度 | 先通用 → 后高质量/领域数据的分阶段策略 |
| **长上下文训练** | 扩展模型上下文窗口（从 4K → 128K → 1M+） | LongRoPE2 (2025)：用 80 倍更少 token 扩展到 128K；Nemotron 3：原生 1M 窗口 |

**团队规模参考**：核心团队 5-15 人，但这是基模团队最"精英"的部分——决策密度最高、单次实验成本最大。

**关键特征**：一次大规模预训练可能耗时数月、消耗数千万美元，因此决策极其谨慎。预训练团队的文化是 "少跑多想"——一年可能只跑 1-3 次正式训练。

## ③ 后训练与对齐（Post-training & Alignment）

**别名**：Fine-tuning / RLHF / Alignment / 对齐团队

**核心职责**：将预训练基座模型转化为安全、有用、符合人类偏好的对话/指令模型。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **SFT（监督微调）** | 指令遵循数据构建、对话数据整理、多轮对话训练 | 高质量 SFT 数据 >> 大量低质量数据已成共识 |
| **奖励模型训练** | 训练 Reward Model（RM），学习人类偏好排序 | 从独立 RM → 模型自身作为 Judge 的趋势 |
| **RLHF/PPO** | 使用 PPO 算法基于 RM 优化模型策略 | 仍在使用但逐步让位给更简单的方法 |
| **DPO/偏好优化** | Direct Preference Optimization，无需独立 RM | DPO 成为 2024 主流，但 2025 又有新变种（KTO, IPO, cDPO） |
| **GRPO** | Group Relative Policy Optimization，DeepSeek 提出 | GRPO-λ (2025)：改进信用分配，AIME24 提升 3-4.5 分；2-GRPO：2 样本即够 |
| **Constitutional AI** | 基于原则的 AI 反馈对齐（RLAIF） | Anthropic 核心方法；2026 论文解释了为什么 RLAIF 有效（潜在价值假说） |
| **安全对齐** | Red teaming、有害内容拒绝、安全边界设定 | HarmBench、SORRY-Bench、TRIDENT：标准化安全评估 |
| **人类标注管理** | 标注团队管理、标注规范设计、标注质量控制 | 高质量标注是瓶颈；Self-Rewarding LM 试图减少对人工标注的依赖 |
| **模型合并** | 将不同能力的微调模型合并为一个统一模型 | Model Merging（TIES, DARE, SLERP）成为实用技术 |
| **多阶段 Pipeline** | 设计 SFT → RM → RL 的完整后训练流水线 | TULU 3：开源的完整后训练 pipeline；Llama 3 后训练报告 |

**团队规模参考**：通常占基模团队 20-30%，是迭代最频繁的团队。

**关键特征**：后训练团队的节奏与预训练完全不同——实验周期短（小时到天级）、迭代频率高（每天多轮实验）、更适合非前沿实验室入局。

## ④ 推理模型训练（Reasoning Model Training）

**别名**：Reasoning / Test-time Compute / o1-like Training / 慢思考

**核心职责**：训练模型在推理时进行深度思考，通过链式推理解决复杂问题。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **推理 RL 训练** | 在思维链（CoT）上进行大规模 RL 训练 | OpenAI o1/o3：RL 训练在 CoT 上；DeepSeek R1：开源复现 |
| **验证器构建** | 为数学/代码/逻辑构建自动验证器作为奖励信号 | RLVR（Reinforcement Learning with Verifiable Rewards）成为主流 |
| **推理数据构建** | 构建高质量推理轨迹数据（问题 + 完整推理链 + 答案） | s1 (2025)：仅 1000 条精选数据即达 o1-preview 水平 |
| **Test-time Compute 控制** | 控制推理时的计算预算（思考多长时间） | Budget Forcing：通过 prompt 控制思考长度 |
| **蒸馏推理能力** | 将大推理模型的能力蒸馏到小模型 | DeepSeek R1-distill 系列：7B/14B/32B 保留推理能力 |
| **多步工具调用推理** | 推理过程中调用代码执行、搜索等工具 | o3/o4-mini 在推理中调用所有 ChatGPT 工具 |

**团队规模参考**：5-10 人的专精团队，2024-2025 年新兴分支，正在从后训练团队中独立出来。

**关键特征**：这是 2024-2025 年最热的方向。推理模型训练介于预训练和后训练之间——需要大规模 RL 训练（类似预训练的资源消耗），但迭代节奏更接近后训练。

## ⑤ 多模态（Multimodal）

**别名**：Vision-Language / 多模态基座 / VLM Team

**核心职责**：让基座模型理解和生成文本之外的模态——图像、视频、音频。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **视觉编码器** | 训练/选择 Vision Encoder（ViT, SigLIP 等） | 视觉数据远比文本"饥饿"，需要更多数据和更大编码器 |
| **视觉-语言对齐** | 将视觉表征与语言模型对齐（projector 训练、跨模态注意力） | VideoLLaMA 3：视觉为中心的训练范式 |
| **图像理解** | 图片理解、OCR、图表解析、文档理解 | 可变分辨率编码成为标配 |
| **视频理解** | 视频理解、时序推理、视频 QA | 视频 token 压缩是关键技术挑战 |
| **图像生成** | 文生图、图像编辑（扩散模型/自回归生成） | Emu3 (Nature 2025)：纯 next-token prediction 统一理解+生成 |
| **视频生成** | 文生视频、视频编辑 | LTX-2：14B 视频 + 5B 音频双流 Transformer |
| **语音/音频** | 语音识别、语音合成、音频理解、实时语音对话 | 豆包实时语音大模型：满意度 4.36 超越 GPT-4o 的 3.18 |
| **全模态统一** | 统一架构处理所有模态（Omni model） | Nexus：语言+音频+视觉三模态统一；MoE 天然产生模态专业化 |
| **多模态数据构建** | 图文对、视频描述、音频转写等多模态训练数据 | 高质量图文数据 >> 海量低质视频数据 |

**团队规模参考**：通常占基模团队 15-25%，是增长最快的部分。

**关键特征**：多模态团队内部往往进一步按模态分组（视觉理解组、生成组、语音组）。2025 年的趋势是 "统一"——用一个模型处理所有模态，而非各搞各的。

## ⑥ 模型架构研究（Architecture Research）

**别名**：Architecture / 架构创新

**核心职责**：设计和改进模型架构，提升效率、能力或扩展性。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **Attention 机制改进** | GQA, MQA, MLA, Sliding Window, Flash Attention | DeepSeek MLA：低维投影减少显存；Flash Attention 3 |
| **MoE 架构** | Mixture of Experts 路由策略、专家数量、负载均衡 | DeepSeek V3：671B 总/37B 活跃；超稀疏 MoE 成趋势 |
| **SSM/Mamba 混合** | State Space Model 与 Transformer 的混合架构 | Nemotron 3 Super (2026)：Mamba+Transformer+MoE，5 倍吞吐 |
| **位置编码** | RoPE 改进、长上下文支持、位置外推 | LongRoPE2：80 倍更少 token 扩展至 128K |
| **归一化方法** | Pre-norm vs Post-norm, RMSNorm, DeepNorm | RMSNorm 成为默认选择 |
| **激活函数** | SwiGLU, GeGLU 等门控激活 | SwiGLU 已成标配 |
| **模型结构搜索** | 自动化探索最优层数/宽度/头数配置 | SEKI (2025)：0.05 GPU-days 完成 NAS |
| **效率架构** | 降低训练/推理成本的架构创新 | Speculative Decoding、Early Exit、稀疏化 |

**团队规模参考**：3-8 人，高度精英化。

**关键特征**：架构创新的节奏与预训练类似——一个好的架构创新可能需要数月研究但影响深远。大多数公司不设独立架构团队，而是由预训练团队核心成员兼任。但少数顶级实验室（DeepSeek、Google）有专门的架构研究小组。

## ⑦ 评估体系（Evaluation）

**别名**：Eval / Benchmarking / 评测团队

**核心职责**：设计、维护和运行评估体系，量化模型能力，指导训练方向。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **标准 Benchmark 维护** | MMLU, MATH, HumanEval, MT-Bench 等标准评估集的运行与管理 | Benchmark 饱和问题日益严重，需要更难的评估 |
| **内部评估集构建** | 构建更贴近真实用例的内部评估集 | OpenHands Index (2026)：5 类真实编码任务评估 |
| **自动化评估 Pipeline** | 端到端自动化评估流水线，支持持续评估 | LM Eval Harness、EleutherAI 评估框架 |
| **LLM-as-Judge** | 使用 LLM 评估其他 LLM 的输出质量 | Arena-Hard：自动化偏好评估；MT-Bench 的 GPT-4 Judge |
| **人类评估** | 组织 human eval、Chatbot Arena 等人类偏好评估 | LMSYS Chatbot Arena 成为事实标准 |
| **安全评估** | Red Teaming、有害输出检测、对抗鲁棒性测试 | HarmBench (510 行为)、AgentHarm (110 恶意任务)、SORRY-Bench (44 话题) |
| **能力细分评估** | 对模型能力进行细粒度画像（数学、代码、推理、知识、创造力等） | 从单一分数 → 多维能力雷达图 |
| **去污染** | 确保评估结果不被训练数据泄漏污染 | 去污染是评估可信度的基础，TULU 3 将其标准化 |
| **竞赛/极端评估** | AIME、Codeforces、SWE-bench 等高难度评估 | o3 在 AIME 2025 达 98.4%；SWE-bench Verified ~53% |

**团队规模参考**：3-8 人，但几乎与所有其他团队都有交集。

**关键特征**：评估团队是 "基模团队的镜子"——他们不直接提升模型能力，但决定了其他团队能否准确知道模型有多好。好的评估团队可以比预训练团队更早发现问题。

## ⑧ 训练基础设施（Training Infrastructure）

**别名**：Infra / Platform / 系统团队

**核心职责**：构建和维护大规模 GPU 集群、分布式训练框架、存储和网络系统。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **分布式训练框架** | 3D 并行（TP+PP+DP）、ZeRO、FSDP 的开发与优化 | Galvatron (2025)：自动最优并行策略；ByteScale：12000+ GPU 长上下文训练 |
| **集群管理** | GPU 集群调度、任务队列、资源分配、故障管理 | K8s + 自研调度器；DeepSeek 在 2048 GPU 上实现高效训练 |
| **通信优化** | AllReduce、NCCL 优化、网络拓扑设计 | DeepSeek 双管道机制：通信与计算并行化 |
| **存储系统** | Checkpoint 存储、数据加载、高速文件系统 | 70B 模型单 checkpoint ~140GB，需高吞吐存储 |
| **容错与恢复** | GPU 故障检测、自动恢复、弹性训练 | MoFa：集成故障容忍建模；大规模训练故障率 ~1-5%/天 |
| **混合精度工程** | FP8/BF16/FP16 混合精度的底层实现与优化 | DeepSeek V3 FP8 训练：零精度损失 |
| **训练性能分析** | GPU 利用率、通信瓶颈、内存分析、性能调优 | 理论峰值利用率 ~50-60% 已属优秀 |
| **硬件适配** | 不同 GPU（A100/H100/H800/国产芯片）的适配和优化 | DeepSeek V3.1：国产芯片优化 |
| **训练监控** | Loss 曲线、梯度范数、硬件状态的实时监控和告警 | 自建 Dashboard + Prometheus/Grafana |

**团队规模参考**：通常占基模团队 15-20%，是最"工程"导向的团队。

**关键特征**：基础设施团队是 "看不见的英雄"——他们的工作决定了其他团队能否高效运转。一个好的 infra 团队可以让同等 GPU 数量产出翻倍。DeepSeek 用 2048 卡训出 V3 的核心竞争力就在基础设施。

## ⑨ 推理部署与模型压缩（Inference & Compression）

**别名**：Serving / Deployment / Inference Optimization / 推理优化

**核心职责**：让训练好的模型高效、低成本地服务于线上请求。

### 子方向

| 子方向 | 具体工作 | 前沿进展 |
|--------|---------|---------|
| **推理框架** | vLLM、TensorRT-LLM、SGLang 等推理引擎的使用和优化 | vLLM (2025)：PagedAttention + Continuous Batching + Prefix Caching |
| **量化** | INT8/INT4/FP8/NF4 权重量化、激活量化 | GPTQ, AWQ, GGUF 各有适用场景 |
| **知识蒸馏** | 大模型 → 小模型的能力转移 | IOA 框架：学生保留 94.7% 教师性能，参数 <1/10 |
| **推理并行** | Tensor Parallelism、Pipeline Parallelism、Speculative Decoding | vLLM PD 分离：Prefill 和 Decode 阶段解耦优化 |
| **KV Cache 优化** | KV Cache 压缩、分页管理、跨请求共享 | PagedAttention 是核心创新；GQA/MQA 天然减少 KV Cache |
| **Speculative Decoding** | 用小模型草拟 → 大模型验证，加速推理 | 主流推理框架已内置支持 |
| **成本优化** | Token 定价策略、PD 分离、弹性伸缩 | 豆包：PD 分离 + 区间定价，成本降 63%，0.0008 元/千 token |
| **端侧部署** | 在手机/边缘设备上运行模型 | Apple Foundation Models (2025)：端侧 3B 模型 |

**团队规模参考**：5-10 人，与基础设施团队紧密协作。

**关键特征**：推理团队的核心 KPI 是 "每 token 成本" 和 "首 token 延迟"。这个团队直接影响产品体验和商业化利润。
