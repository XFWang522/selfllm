# SelfLLM 技术路线文档

## 1. 项目定位与目标

### 1.1 终极目标

构建一个**完全自主的 LLM 自我进化闭环系统**，使大语言模型能够：

1. **自主编写**自己的训练代码（包括数据处理、模型架构调整、训练循环、优化器配置等）
2. **自主搜集**训练数据（联网搜索、合成生成、数据清洗与筛选）
3. **自主执行**训练流程（分布式训练调度、资源管理、容错恢复）
4. **自主评估**训练结果（设计评估方案、运行 benchmark、分析弱点）
5. **自主迭代**训练策略（根据评估结果修改代码、调整数据、优化超参）

### 1.2 阶段性目标

| 阶段 | 时间 | 目标 | 验收标准 |
|------|------|------|---------|
| P0 原型验证 | M1-M4 | 在受限场景下验证闭环可行性 | 模型在特定 benchmark 上通过自训练提升 ≥10% |
| P1 单能力闭环 | M5-M8 | 实现单一能力（如数学/代码）的完整闭环 | 数学推理能力自主提升至 SOTA 水平的 90% |
| P2 多能力闭环 | M9-M14 | 扩展到多能力同时自主优化 | 综合 benchmark 得分自主提升 ≥15% |
| P3 开放域进化 | M15-M18 | 开放域数据搜集 + 自主策略发现 | 发现人类未曾设计过的训练策略 |
| P4 持续进化 | M19-M24 | 稳定运行的持续自进化系统 | 连续运行 30 天，性能持续上升无退化 |

### 2.2 五大核心模块详解

#### ① Code Generator（代码生成器）

**职责**：LLM 自主编写、修改和优化训练相关代码。

**技术方案**：
- 基于 AI Coding Agent（OpenHands/Cursor Agent）构建代码生成能力
- 维护一个 **代码仓库（Training Code Repository）**，所有训练代码通过 git 版本控制
- 每次迭代产生一个新的代码分支，记录完整的修改历史
- 代码生成前需通过 **静态分析 + 单元测试 + 沙箱运行** 三重验证

**输出物**：
- 数据处理 pipeline 代码
- 模型架构定义/修改代码
- 训练循环代码（支持 Megatron-LM / DeepSpeed）
- 超参数配置文件
- 评估脚本

**关键设计**：
```python
class CodeGenerator:
    def __init__(self, base_model, coding_agent):
        self.model = base_model          # 当前最佳模型
        self.coder = coding_agent        # AI Coding Agent
        self.repo = TrainingCodeRepo()   # Git 管理的代码仓库
        self.sandbox = CodeSandbox()     # 安全沙箱

    def generate_training_code(self, strategy: TrainingStrategy) -> CodeArtifact:
        """根据训练策略生成完整的训练代码"""
        # 1. 分析当前策略需求
        requirements = self.model.analyze_strategy(strategy)
        # 2. AI Coder 编写代码
        code = self.coder.write(requirements, context=self.repo.history)
        # 3. 沙箱验证
        test_result = self.sandbox.validate(code)
        # 4. 代码审查（用模型自己审查）
        review = self.model.review_code(code, test_result)
        # 5. 迭代修复
        while not review.approved:
            code = self.coder.fix(code, review.feedback)
            test_result = self.sandbox.validate(code)
            review = self.model.review_code(code, test_result)
        return code
```

#### ② Data Collector（数据搜集器）

**职责**：LLM 自主决定需要什么数据，并通过联网搜索、合成生成等方式获取。

**数据来源分层**：

| 层级 | 来源 | 获取方式 | 适用场景 |
|------|------|---------|---------|
| L0 | 自身生成 | Self-play / 合成 | 推理、代码、对话 |
| L1 | 公开数据集 | API / 下载 | 通用知识、benchmark 数据 |
| L2 | 网页爬取 | 搜索引擎 + 爬虫 | 时事、领域知识 |
| L3 | 学术论文 | arXiv / Semantic Scholar | 前沿研究知识 |
| L4 | 代码仓库 | GitHub API | 编程能力 |
| L5 | 交互反馈 | 在线服务日志 | 真实用户需求 |

**核心技术**：
- **需求分析引擎**：模型分析自身弱点，确定需要什么类型的数据
- **智能搜索 Agent**：使用 Web Browser Agent 在互联网上定向搜索
- **数据质量评估**：模型自己对搜集到的数据进行质量打分（参考 Self-Rewarding LM）
- **去重与去污染**：确保新数据不与评估集重叠
- **版权合规检查**：自动识别并排除受版权保护的内容

**数据搜集策略的自主演化**：
```
Round 1: 模型发现数学推理弱 → 搜集数学题库 + 合成数学推理链
Round 2: 模型发现代码能力弱 → 爬取 GitHub 高质量代码 + 生成代码题
Round 3: 模型发现长文本理解弱 → 搜集长文档 + 构建长文本 QA
Round N: 模型自主发现新的弱点并制定搜集策略
```

#### ③ Train Executor（训练执行器）

**职责**：执行由 Code Generator 生成的训练代码，管理分布式训练资源。

**技术栈**：
- 底层框架：Megatron-LM + DeepSpeed ZeRO-3
- 集群管理：Kubernetes + GPU 调度器
- 断点续训：自动 checkpoint + 故障恢复
- 混合精度：BF16/FP8 自动选择

**执行流程**：
1. 接收 Code Generator 输出的训练代码和 Data Collector 输出的数据
2. 自动配置分布式训练环境（tensor parallelism, pipeline parallelism, data parallelism）
3. 启动训练任务，实时监控 loss 曲线、梯度范数、显存使用
4. 异常检测：loss spike、NaN 梯度、OOM 等自动处理
5. 训练完成后输出模型 checkpoint 和训练日志

**安全沙箱设计**：
- 训练代码在隔离容器中运行
- 网络访问白名单机制
- GPU 资源配额限制
- 自动超时和 kill 机制

#### ④ Eval Engine（评估引擎）

**职责**：LLM 自主设计评估方案、运行评估、分析结果。

**评估维度**：

| 维度 | 评估方法 | 自动化程度 |
|------|---------|-----------|
| 数学推理 | MATH, GSM8K, AMC/AIME | 完全自动（答案可验证） |
| 代码能力 | HumanEval, SWE-bench, MBPP | 完全自动（测试用例验证） |
| 通用知识 | MMLU, ARC, HellaSwag | 完全自动（选择题） |
| 指令遵循 | IFEval, MT-Bench | 半自动（LLM-as-Judge） |
| 长文本 | RULER, LongBench | 完全自动 |
| 安全性 | TruthfulQA, BBQ | 半自动 |
| 创造力 | 模型自主设计评估 | LLM 评估 |

**关键创新——自适应评估**：
- 模型不仅在固定 benchmark 上评估，还能**自主设计新的评估场景**
- 根据训练策略的目标，动态选择最相关的评估集
- 生成细粒度的能力画像，而不仅仅是分数
- 对比前后版本的能力变化，定位提升与退化

#### ⑤ Strategy Optimizer（策略优化器）

**职责**：根据评估结果，自主决定下一轮训练策略的调整方向。

**策略空间**：

```
Strategy = {
    training_method:    [SFT, DPO, RLHF, GRPO, STaR, Self-Play, ...],
    data_composition:   {math: 0.3, code: 0.2, reasoning: 0.2, ...},
    learning_rate:      [1e-5, 5e-5, 1e-4, ...],
    batch_size:         [256, 512, 1024, ...],
    epochs:             [1, 2, 3, ...],
    architecture_mods:  [LoRA, full_finetune, MoE_routing, ...],
    data_augmentation:  [paraphrase, backtranslation, self_play, ...],
    curriculum:         [easy_to_hard, hard_to_easy, mixed, adaptive, ...],
    loss_function:      [cross_entropy, DPO_loss, KTO_loss, custom, ...],
}
```

**优化方法**：
1. **经验回放**：维护所有历史实验的 (策略, 结果) 记录
2. **LLM 推理**：用模型自身分析 "为什么上一轮策略效果不好"
3. **进化搜索**：对历史最优策略进行变异和组合（类似 FunSearch）
4. **贝叶斯优化**：在策略空间中用高斯过程指导搜索
5. **元学习**：从策略-结果的历史中学习 "学习如何学习"

### 2.3 AI Coding 深度集成

AI Coding 不仅仅是一个辅助工具，而是系统的**核心能力层**：

```
传统方式：人写代码 → 跑训练 → 人看结果 → 人改代码 → 循环
SelfLLM ：LLM 写代码 → 跑训练 → LLM 看结果 → LLM 改代码 → 自动循环
```

**集成方式**：

| 环节 | AI Coding 的作用 |
|------|-----------------|
| 训练代码编写 | 根据策略自动生成完整训练脚本 |
| 数据处理 Pipeline | 编写数据下载、清洗、格式转换代码 |
| 评估脚本 | 根据评估需求编写自定义评估代码 |
| Bug 修复 | 训练出错时自动 debug 并修复 |
| 性能优化 | 分析训练瓶颈并优化代码性能 |
| 新方法实现 | 将论文中的新方法翻译为可执行代码 |
| 基础设施代码 | 编写集群管理、任务调度相关代码 |

**AI Coding Agent 的能力要求**：
- 理解 PyTorch / JAX / Megatron-LM 等深度学习框架
- 能编写分布式训练代码（DDP, FSDP, tensor parallel）
- 能阅读和理解训练日志，诊断训练问题
- 能将自然语言描述的策略转化为代码变更
- 具备 Code Review 能力，能审查自己和历史版本的代码

## 3. 技术路线详细规划

### Phase 0：原型验证（M1-M4）

**目标**：在受限场景下验证 "LLM 写代码训练自己" 的基本可行性

**M1-M2：基础框架搭建**
- [ ] 搭建 Code Generator 原型，基于 OpenHands + 基座模型
- [ ] 构建安全沙箱环境，限制训练代码的权限
- [ ] 实现基本的训练代码模板库（SFT, DPO 基础模板）
- [ ] 搭建小规模 GPU 集群（64-128 卡）用于验证实验
- [ ] 实现基础评估流水线（MATH, HumanEval, MMLU）

**M3-M4：闭环原型**
- [ ] 选定一个窄领域（如数学推理）作为试点
- [ ] 实现完整的 "生成代码 → 搜集数据 → 训练 → 评估 → 调整" 单次循环
- [ ] 验证模型能否生成可运行的训练代码
- [ ] 验证模型能否根据评估结果合理调整策略
- [ ] 运行 3-5 轮自主迭代，观察性能曲线

**里程碑**：GSM8K 准确率通过自主训练提升 ≥10%

### Phase 1：单能力闭环（M5-M8）

**目标**：在数学推理或代码生成上实现稳定的自主进化闭环

**M5-M6：强化代码生成能力**
- [ ] 训练专用的 Code Generator 模型（在训练代码上微调）
- [ ] 实现代码变更的 diff 级管理和回滚
- [ ] 引入 Code Review Agent，提升代码质量
- [ ] 实现训练代码的自动化测试框架

**M7-M8：强化数据搜集与策略优化**
- [ ] 接入搜索引擎 API 和学术数据库
- [ ] 实现 Self-Play 数据生成（如 Language Self-Play）
- [ ] 构建策略优化的经验回放系统
- [ ] 实现自适应课程学习（从易到难自动调整）

**里程碑**：
- MATH benchmark 通过自主训练达到 SOTA 90%
- 生成的训练代码通过人工审查，质量评分 ≥4/5
- 完成 ≥20 轮自主迭代无需人工干预

### Phase 2：多能力闭环（M9-M14）

**目标**：将闭环扩展到多个能力维度，解决能力冲突问题

**M9-M10：多能力并行训练**
- [ ] 实现多目标优化框架（同时优化数学、代码、推理、知识）
- [ ] 解决灾难性遗忘问题（旧能力退化）
- [ ] 构建能力平衡监控和自动修复机制
- [ ] 实现 MoE（Mixture of Experts）架构的自主调整

**M11-M12：开放域数据搜集**
- [ ] 扩展 Web Browsing Agent 的搜索范围
- [ ] 实现多语言数据搜集
- [ ] 构建数据质量的多维评估体系
- [ ] 实现增量训练数据管理（新数据 + 原始数据的混合）

**M13-M14：策略空间扩展**
- [ ] 允许模型自主提出新的训练方法（不限于预定义方法集）
- [ ] 实现训练方法的 A/B 测试框架
- [ ] 引入模型架构搜索（NAS）的自主化
- [ ] 构建长期策略记忆和知识库

**里程碑**：
- 综合 benchmark（MMLU + MATH + HumanEval + MT-Bench）平均提升 ≥15%
- 模型自主发现并实现至少 1 个新的训练技巧
- 无灾难性遗忘，所有能力维度均有提升

### Phase 3：开放域进化（M15-M18）

**目标**：模型能够自主发现训练目标、设计评估方案、创造新方法

**核心突破点**：
- [ ] 模型自主提出 "我应该学习什么新能力"
- [ ] 模型自主设计评估方案而不依赖固定 benchmark
- [ ] 模型自主发现训练数据中的分布缺陷并修正
- [ ] 实现跨领域知识迁移的自主化
- [ ] 构建模型自我认知能力（知道自己不知道什么）

### Phase 4：持续进化系统（M19-M24）

**目标**：构建可长期稳定运行的自进化系统

**核心挑战**：
- [ ] 防止长期运行的性能退化（model collapse）
- [ ] 实现与真实世界知识的持续同步
- [ ] 构建安全约束下的自由进化机制
- [ ] 系统的自监控、自修复、自扩展

## 4. 关键技术选型

### 4.1 基座模型选择

| 候选 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| Llama 4 (开源) | 完全可控、可修改、社区强 | 可能不是最强 | ⭐⭐⭐⭐⭐ |
| Qwen 3 (开源) | 中文能力强、多模态 | 许可证限制 | ⭐⭐⭐⭐ |
| Mistral Large | 欧洲合规、效率高 | 规模限制 | ⭐⭐⭐ |

**推荐**：以 Llama 4 70B/405B 为主基座，Qwen 3 为辅助验证。开源模型是必须的——我们需要完全控制模型权重和训练流程。

### 4.2 训练框架

- **主框架**：Megatron-LM（支持 3D 并行，大规模训练稳定）
- **辅助**：DeepSpeed ZeRO-3（灵活性高，适合快速实验）
- **微调**：PEFT/LoRA（用于快速迭代实验）
- **RL 训练**：TRL + vLLM（PPO/DPO/GRPO 训练）

### 4.3 AI Coding 框架

- **主力**：OpenHands CodeAct（开源、SWE-bench SOTA）
- **辅助**：自建 Coding Agent（基于基座模型微调的专用代码生成器）
- **代码管理**：Git + 自动化 CI/CD pipeline

### 4.4 数据搜集框架

- **Web 搜索**：Serper API / SearxNG（搜索引擎访问）
- **网页解析**：Playwright + BrowserAgent（网页浏览和提取）
- **学术文献**：Semantic Scholar API + arXiv API
- **代码数据**：GitHub API + The Stack v2

## 6. 风险管理

### 6.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Model Collapse | 高 | 致命 | 保留原始数据混合、多样性监控 |
| 代码生成质量不够 | 中 | 高 | 沙箱验证 + 人工审查兜底 |
| 训练不稳定 | 中 | 中 | 自动 checkpoint + 异常检测 |
| 能力退化 | 高 | 高 | 能力画像监控 + 自动修复 |
| 评估不准确 | 中 | 高 | 多维度交叉验证 |

### 6.2 安全风险

| 风险 | 缓解措施 |
|------|---------|
| 模型生成有害训练代码 | 代码沙箱 + 权限隔离 + 人工审查 |
| 数据投毒 | 数据来源审计 + 质量过滤 |
| 能力失控 | 能力上限阈值 + 熔断机制 |
| 对齐漂移 | 持续安全评估 + Constitutional AI 约束 |
